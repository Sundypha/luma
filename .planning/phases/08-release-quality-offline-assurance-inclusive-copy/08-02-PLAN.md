<objective>
Eliminate the initial-load spinner flash on Home and Calendar screens by seeding ViewModels with pre-fetched data, and verify all interactions meet the "feels immediate" bar.

Purpose: NFR-01 requires common screens and logging actions to feel immediate. Both HomeScreen and CalendarScreen show a `CircularProgressIndicator` until `hasInitialEvent` becomes true from the async Drift stream. Since `main()` already awaits `repository.listOrderedByStartUtc()` before `runApp()`, the data is available — but it's not threaded to the ViewModels. This plan closes that gap.

Output: ViewModels render immediately with pre-fetched data; no spinner visible on normal app startup; all animations within 200–300ms spec.
</objective>

  <action>
Thread pre-fetched period data from `main()` through to both ViewModels so they start with `hasInitialEvent = true` and never show the loading spinner on normal startup.

**Step 1: Fetch initial data in main()**

In `main()`, after the existing `listOrderedByStartUtc()` call (line ~26), also fetch the full `StoredPeriodWithDays` list that ViewModels need:

```dart
final periodsWithDays = await repository.watchPeriodsWithDays().first;
```

This awaits a single emission from the stream, which is a lightweight SQLite query and adds negligible startup time (sub-millisecond on warm DB).

**Step 2: Thread through LumaApp → TabShell**

Add an optional `List<StoredPeriodWithDays>? initialPeriodsWithDays` parameter to `LumaApp` and `TabShell`. Pass it from `main()` through `LumaApp` to `TabShell`.

In `LumaApp`: add `this.initialPeriodsWithDays` field, pass to `TabShell(initialPeriodsWithDays: widget.initialPeriodsWithDays, ...)`.

In `TabShell`: add `final List<StoredPeriodWithDays>? initialPeriodsWithDays;` constructor parameter, pass to ViewModel constructors.

**Step 3: Add optional initial data to HomeViewModel**

Add a named `{List<StoredPeriodWithDays>? initialData}` parameter to the `HomeViewModel` constructor. If non-null, call `_onData(initialData)` before subscribing to the stream. This sets `_hasInitialEvent = true` immediately:

```dart
HomeViewModel(this._repository, this._calendar, {
  List<StoredPeriodWithDays>? initialData,
}) {
  if (initialData != null) {
    _onData(initialData);
  }
  _subscription = _repository.watchPeriodsWithDays().listen(
    _onData,
    onError: _onStreamError,
  );
}
```

**Step 4: Add optional initial data to CalendarViewModel**

Same pattern as HomeViewModel:

```dart
CalendarViewModel(this._repository, this._calendar, {
  List<StoredPeriodWithDays>? initialData,
}) {
  if (initialData != null) {
    _onData(initialData);
  }
  _subscription = _repository.watchPeriodsWithDays().listen(
    _onData,
    onError: _onStreamError,
  );
}
```

**Step 5: Update TabShell to pass initial data**

In `_TabShellState.initState()`:
```dart
_calendarVm = CalendarViewModel(widget.repository, widget.calendar,
    initialData: widget.initialPeriodsWithDays);
_homeVm = HomeViewModel(widget.repository, widget.calendar,
    initialData: widget.initialPeriodsWithDays);
```

**Important constraints:**
- Keep `initialData` optional so widget tests that don't provide it still work (VMs fall back to stream-only behavior).
- The stream subscription still runs — the first stream event will replace the seed data, which is correct (keeps data reactive).
- Do NOT call `notifyListeners()` from the constructor when seeding — the ViewModel isn't attached to a listener yet. The `_onData` call already sets `_hasInitialEvent = true`, so the first `ListenableBuilder` build will see data immediately.

**Step 6: Verify animation/transition durations**

Scan `apps/ptrack/lib/` for `Duration` and `AnimatedSize` usages. Confirm all transition durations are within the 200–300ms spec from CONTEXT.md. The research found `AnimatedSize` at 200ms and page transitions at 300ms — confirm these are still correct.

Also verify all 4 custom painters in `calendar_painters.dart` have correct `shouldRepaint` implementations (research confirmed they do — just verify none were modified since).
