  <action>
Audit every `IconButton`, `Icon` used as a tap target, and `FloatingActionButton` across all UI files for proper accessibility labels.

**CONTEXT.md rule:** "Icon + text for primary actions. Icon-only acceptable for well-known patterns (back, close, delete) with accessibility labels."

**Audit checklist for each widget:**

1. `IconButton` — must have `tooltip` parameter set (Flutter's `IconButton.tooltip` sets both the visual tooltip and semantic label). AppBar back buttons are automatically handled by Flutter.
2. `FloatingActionButton` — must have `tooltip` parameter set (TabShell FAB already has `tooltip: marked ? 'Add symptoms' : 'Mark today'` ✓).
3. Icons inside `TextButton.icon` / `FilledButton.icon` — these have text labels so are accessible by default ✓.
4. Standalone `Icon` widgets used decoratively (e.g., in cards) — do not need labels.

**For each IconButton found without tooltip:**
- Add a descriptive `tooltip:` that matches the action (e.g., `tooltip: 'Go to today'` for a today button).
- Use verbs for action tooltips ("Go to today", "Close", "Delete") per CONTEXT.md.

**Known patterns to check:**
- Calendar "Today" button (TextButton.icon — should be fine)
- Header chevrons in table_calendar (handled by the library — verify)
- AppBar back buttons on pushed screens (handled by MaterialApp — verify)
- Lock screen biometrics button (text label present — verify)
- Drawer hamburger (handled by Scaffold.drawer — verify)

Only add tooltips where genuinely missing. Do NOT add tooltips to text+icon combinations (CONTEXT.md: "No tooltips — if a label needs a tooltip, the label is wrong" applies to text labels, not icon-only accessibility).
