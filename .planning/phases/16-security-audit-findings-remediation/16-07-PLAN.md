  <name>Task 1: Encrypted auto-backup with internal key from secure storage</name>
  <files>packages/ptrack_data/lib/src/export/backup_service.dart, packages/ptrack_data/lib/src/export/export_service.dart</files>
  <action>
  **Backup encryption key** (in `backup_service.dart` or a new `backup_encryption_key.dart`):

  1. Create a function `getOrCreateBackupEncryptionKey`:
     - Storage key: `ptrack_backup_encryption_key`
     - Generates a strong random passphrase (32 bytes, base64-encoded — producing ~43 chars, well above the 8-char minimum from Plan 16-04).
     - Stores in `FlutterSecureStorage`.
     - Returns the passphrase string for use with `ExportOptions(password: ...)`.
     - Add `deleteBackupEncryptionKey` for factory reset cleanup.

  2. This key is SEPARATE from the SQLCipher DB encryption key (Plan 16-05). Different key for different purpose: DB key is hex for PRAGMA, backup key is a passphrase for Argon2id KDF.

  **BackupService changes** (`backup_service.dart`):

  3. Add `FlutterSecureStorage? storage` parameter to `BackupService` constructor (for testability — defaults to `const FlutterSecureStorage()`).

  4. In `createBackup()`:
     - Retrieve (or generate) the backup key: `final backupKey = await getOrCreateBackupEncryptionKey(storage: _storage);`
     - Change the export call to include the password:
       ```dart
       final export = await ExportService(_db).exportData(
         options: ExportOptions.everything(password: backupKey),
       );
       ```
     - The existing `_finish` method in `ExportService` already encrypts when `password != null`, so this change alone makes backups encrypted.

  5. Add a `Future<bool> isBackupEncryptionEnabled()` convenience method that returns `true` (it's always enabled now). This can be used by the UI to show encryption state.

  **Import compatibility for internal backups**:

  6. The import flow in `ImportService` already handles encrypted files by asking for a password. For internal auto-backups, the password is the stored key. The app UI will need to retrieve the backup key when restoring an auto-backup and pass it to the import flow.

  7. Add `Future<String?> getBackupEncryptionKey({FlutterSecureStorage? storage})` — returns the stored key or null if not yet created. This is for the restore path.

  8. In `BackupService`, add a convenience method:
     ```dart
     Future<LumaExportData> decryptBackup(Uint8List bytes) async {
       final key = await getOrCreateBackupEncryptionKey(storage: _storage);
       final importService = ImportService(_db, calendar: ...);
       // This needs the calendar context — consider accepting it as a parameter
       // or making decryptBackup a standalone helper.
     }
     ```
     Actually, simpler approach: let the existing import UI retrieve the key and pass it. Don't add a new method. Just document that auto-backup restore should call `getBackupEncryptionKey()` and pass it as the password.

  **Factory reset cleanup**:

  9. In `main.dart` `_resetApp()` (if not already added by Plan 16-03), call `deleteBackupEncryptionKey()` alongside `deleteDbEncryptionKey()` and `deleteAllBackups()`. All key material and backup files are removed on reset.
