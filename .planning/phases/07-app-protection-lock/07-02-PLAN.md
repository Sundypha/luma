    Constructor:
    ```dart
    const PinEntryWidget({
      super.key,
      required this.pinLength,         // e.g. 4 or 6; dots shown = pinLength
      required this.onSubmit,          // void Function(String pin)
      this.onChanged,                  // void Function(String current)?
      this.errorText,                  // shown below dots when non-null
      this.submitOnComplete = true,    // auto-submit when all dots filled
    })
    ```

    Internally manages a `String _current = ''`. Layout: Row of filled/empty dots (●/○), then a 3×4 grid of keys (1–9, then [backspace], 0, [ok]). Use Material `IconButton`/`TextButton` for keys. Show backspace as `Icons.backspace_outlined`; show ok button only when `!submitOnComplete` or as a visual affordance. Emit `onChanged` on each digit change, `onSubmit` when length == pinLength (if submitOnComplete) or ok tapped.

    Keep pin length configurable — default 4. Project will use 4-digit PIN (Claude discretion).

    **lock_view_model.dart** — `ChangeNotifier` managing unlock attempt state.

    Constructor:
    ```dart
    LockViewModel({required this.lockService})
    ```

    State:
    - `bool isLoading` — true during async verify/biometric call
    - `bool hasError` — true after a failed PIN or biometric attempt
    - `String? errorMessage` — e.g. "Incorrect PIN" or "Biometric failed"
    - `int attemptCount` — incremented on each wrong PIN attempt (for future cooldown)

    Methods:
    - `Future<void> verifyPin(String pin, {required VoidCallback onUnlocked})` — set loading, call `lockService.verifyPin(pin)`, on true call `onUnlocked()`, on false set hasError + errorMessage "Incorrect PIN", always clear loading.
    - `Future<void> authenticateBiometric({required VoidCallback onUnlocked})` — set loading, call `lockService.authenticateWithBiometrics('Authenticate to unlock Luma')`, on true call `onUnlocked()`, on false set hasError + errorMessage null (biometric failure is self-explanatory), clear loading.
    - `void clearError()` — reset hasError/errorMessage/loading.

    **pin_setup_sheet.dart** — Multi-step modal bottom sheet for first-time lock setup.

    Entry point: `Future<bool> showPinSetupSheet(BuildContext context, {required LockService lockService})`
    Returns `true` if setup completed fully; `false` if cancelled at any step.

    Steps implemented as a `StatefulWidget` inside `showModalBottomSheet`:
    - **Step 0 — Acknowledgment**: Show a short paragraph explaining: "App lock encrypts access with a PIN. If you forget your PIN, the only recovery option is a data reset — your period data will be erased. Export your data regularly to avoid data loss." Show `[I understand, continue]` button and `[Cancel]` button. Proceeding to Step 1 only after tapping continue.
    - **Step 1 — Create PIN**: Show `PinEntryWidget(pinLength: 4, submitOnComplete: true, onSubmit: _onFirstEntry)`. Title: "Create a PIN". Instructions: "Enter a 4-digit PIN."
    - **Step 2 — Confirm PIN**: Show `PinEntryWidget(pinLength: 4, submitOnComplete: true, onSubmit: _onConfirmEntry)`. Title: "Confirm your PIN". If mismatch: show `errorText: 'PINs do not match'` and clear `PinEntryWidget` to re-enter. On match: call `lockService.createPin(pin)` then `lockService.enableLock()`, then proceed to Step 3.
    - **Step 3 — Biometric offer** (only if `await lockService.canUseBiometrics()` is true): "Enable biometrics for faster unlock?" Show `[Enable biometrics]` and `[Skip]` buttons. Tapping enable calls `lockService.setBiometricsEnabled(true)`.

    On dismissing the sheet via back navigation or tapping outside at any step before completion: return `false` (use `WillPopScope` or `PopScope` if available — check Flutter SDK version; or use `showModalBottomSheet` `isDismissible: false` + explicit cancel buttons so the result is always explicit). Prefer explicit cancel buttons over `isDismissible: false` for usability.

    Lock only becomes enabled when Step 2 completes (i.e., `lockService.enableLock()` is only called inside the success path of Step 2). If user cancels before that, no enableLock call is made.

    Entry point: `Future<void> showForgotPinSheet(BuildContext context, {required VoidCallback onReset})`

    1. **Heading**: "Forgot your PIN?"
    2. **Body copy** (required by LOCK-03 — honest limitations, no false recovery channel):

       ```
       Before resetting, export your data from Data settings so you can restore it afterwards.
       Resetting will erase all period and symptom history from this device.
       ```
    3. Two buttons:
       - `TextButton('Cancel')` — pops sheet without calling onReset.
       - `FilledButton(style: red/error color, 'Erase all data and reset')` — pops sheet and calls `onReset()`.

    Copy must NOT include any phrase implying a server-side recovery channel ("contact support", "recover via email", "reset from cloud"). This is checked by LOCK-03.

    Constructor:
    ```dart
    class LockSettingsScreen extends StatefulWidget {

        super.key,
    ```

    UI layout (`Scaffold` with back navigation, title "App lock"):
    - **When lock is OFF**: `SwitchListTile` "App lock" subtitle: "Lock with PIN or biometrics when returning from background." `onChanged`: calls `_enableLock()`.
    - **When lock is ON**: Same `SwitchListTile` but value=true; turning off calls `_disableLock()`.

    Below switch when lock is ON:
    - `ListTile` "Change PIN" — calls `_changePIN()`.
    - If `_canUseBiometrics`: `SwitchListTile` "Use biometrics" — toggles `lockService.setBiometricsEnabled`.
    - `ListTile` "Lock now" with trailing `IconButton(Icons.lock_outlined)` — calls `lockService` is not the right layer for "Lock now" from here; instead, expose an `onLockNow` callback from `LockSettingsScreen` OR have the screen call back to the parent. Simplest: add `required VoidCallback onLockNow` to the constructor. When tapped: calls `onLockNow()` (which will be wired in 07-03 to trigger `LockGate._isLocked = true`).

    Re-auth flows:
    - `_disableLock()`: show a mini re-auth dialog (or push an inline re-auth step). Simplest consistent approach: show `showDialog` with `PinEntryWidget` (4 digit) + biometric button if available. On correct PIN → call `lockService.disableLock()`, refresh UI. On cancel → no action.
    - `_changePIN()`: same re-auth dialog → on success, call `showPinSetupSheet` skipping the ack step (jump directly to PIN creation step). Simplest: add an optional `skipAck: bool` parameter to `showPinSetupSheet`. On completion of new PIN, call `lockService.createPin(newPin)` (which overwrites the stored hash).
    - `_enableLock()`: call `showPinSetupSheet(context, lockService: lockService)`. If returns true: refresh `_isEnabled` from `lockService.isEnabled`. If returns false: no change.

    Note: "Lock now" action is wired in Plan 07-03 when `LockGate` is added.
