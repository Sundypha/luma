  <action>
    **SymptomFormViewModel** (`ChangeNotifier`):
    - Constructor takes `PeriodRepository`, required `DateTime day` (UTC midnight), required `int periodId`, optional `StoredDayEntry? existing` (pre-fill for edit).
    - State fields:
      - `FlowIntensity? flowIntensity` (initialized from existing)
      - `PainScore? painScore` (initialized from existing)
      - `Mood? mood` (initialized from existing)
      - `String notes` (initialized from existing or '')
      - `bool isSaving = false`
      - `String? errorText`
      - `bool get isEditing` → existing != null
    - Setter commands that call notifyListeners: `setFlow(FlowIntensity?)`, `setPain(PainScore?)`, `setMood(Mood?)`, `setNotes(String)`.
    - `Future<bool> save()`:
      1. Set `isSaving = true`, `errorText = null`, notifyListeners.
      2. Build `DayEntryData(dateUtc: day, flowIntensity: ..., painScore: ..., mood: ..., notes: notes.isEmpty ? null : notes)`.
      3. If editing: call `repository.updateDayEntry(existing!.id, data)`.
      4. If creating: call `repository.upsertDayEntryForPeriod(periodId, data)`.
      5. On success: `isSaving = false`, notifyListeners, return true.
      6. On error: `isSaving = false`, `errorText = e.toString()`, notifyListeners, return false.
    - `Future<bool> clearSymptoms()`:
      1. If existing is null, return false.
      2. Call `repository.deleteDayEntry(existing!.id)`.
      3. Return true on success.
    - dispose: call super.dispose() (no subscriptions to cancel).

    **SymptomFormSheet** (widget):
    - A `StatefulWidget` that creates/disposes `SymptomFormViewModel` internally (per Research Pitfall 7 — sheet has its own lifecycle).
    - Constructor: `required PeriodRepository repository`, `required DateTime day`, `required int periodId`, `StoredDayEntry? existing`.
    - In `initState`: create VM.
    - In `dispose`: call VM dispose.
    - UI (using `ListenableBuilder` on VM):
      - Title: "Add symptoms" or "Edit symptoms" based on `vm.isEditing`
      - Flow intensity: `SegmentedButton<FlowIntensity?>` with null + 3 values (Light/Medium/Heavy)
      - Pain score: `SegmentedButton<PainScore?>` with null + 5 values (use compactLabel)
      - Mood: `Wrap` of `ChoiceChip` widgets for each Mood value (emoji + label)
      - Notes: `TextField` with maxLines: 3
      - Save button: `FilledButton` disabled when `vm.isSaving`
      - Cancel button: `TextButton` that pops
      - If editing: "Clear symptoms" `TextButton` that calls `vm.clearSymptoms()` then pops
      - Error text shown if `vm.errorText` non-null
    - No date picker. No period lifecycle controls. The date is fixed to the constructor parameter.

    **Top-level helper:**
    ```dart
    Future<void> showSymptomFormSheet(
      BuildContext context, {
      required PeriodRepository repository,
      required DateTime day,
      required int periodId,
      StoredDayEntry? existing,
    })
    ```

    **Unit tests** for SymptomFormViewModel:
    - Initial state: fields reflect existing (or defaults if null)
    - Setters update state and notify listeners
    - `save()` creates: calls upsertDayEntryForPeriod with correct args
    - `save()` edits: calls updateDayEntry with existing id
    - `save()` sets isSaving during operation
    - `clearSymptoms()` calls deleteDayEntry

  <action>
    **CalendarScreen routing change:**
    In `_openDayDetail` (or the equivalent after Plan 03's refactor), remove the blank-day bypass that skips to `showLoggingBottomSheet`. Every tap now calls `showDayDetailSheet`. Remove the conditional `if (!dayData.hasLoggedData && !dayData.isPredictedPeriod && dayData.loggedPeriodState == PeriodDayState.none)` branch. All days go to day detail.

    **DayDetailSheet rewrite:**
    Replace the entire current implementation. The new sheet receives:
    - `DateTime selectedDay`
    - `CalendarViewModel viewModel` (live data, not stale snapshot)
    - `PeriodRepository repository`

    The sheet uses `ListenableBuilder` on the CalendarViewModel so it rebuilds when data changes (e.g., after marking a day, the sheet reflects the update).

    **Day state determination** (compute from viewModel's live data each build):
    - `isFuture`: day is after today
    - `isPredicted`: viewModel.dayDataMap[dayNorm]?.isPredictedPeriod == true
    - `isOnPeriod`: viewModel.dayDataMap[dayNorm]?.loggedPeriodState != PeriodDayState.none
    - `hasDayEntry`: lookup day entry from viewModel.periodsWithDays
    - Find `StoredPeriodWithDays?` covering this day (for period ID, day entries)

    **Render based on state (from INVESTIGATION.md §3.4):**

    1. **Empty day (today or past):** Title + date. Single button: "I had my period" → calls `repository.markDay(day)` → pop sheet.

    2. **Empty day (future):** Title + date. Text: "Future dates — check back when this day arrives." No buttons.

    3. **Predicted day (today or past):** Prediction info card (reuse existing card design). Button: "I had my period" → `repository.markDay(day)` → pop.

    4. **Predicted day (future):** Prediction info card. Text: "You can log this once the day arrives." No buttons.

    5. **Period day — no symptoms (today or past):**
       - Period day number (e.g., "Period day 3")
       - "Add symptoms" → `showSymptomFormSheet(context, repository: ..., day: dayNorm, periodId: pwd.period.id)` → pop first, then show form
       - "Remove this day" → call `repository.unmarkDay(day)` → pop. If day has entry, show warning first.
       - "Delete entire period" → confirmation dialog → `repository.deletePeriod(pwd.period.id)` → pop.

    6. **Period day — with symptoms (today or past):**
       - Show logged data (flow, pain, mood, notes) in chip rows (reuse `_chipRow` pattern)
       - Period day number
       - "Edit" → `showSymptomFormSheet(context, repository: ..., day: ..., periodId: ..., existing: entry)` → pop first
       - "Clear symptoms" → `repository.deleteDayEntry(entry.id)` (day stays marked, entry removed) → rebuild via ListenableBuilder
       - "Remove this day" → warning dialog (losing symptoms) → `repository.unmarkDay(day)` → pop
       - "Delete entire period" → confirmation → `repository.deletePeriod(...)` → pop

    **Remove swipe/PageView navigation.** The day detail sheet shows one day at a time. Adjacent-day swipe was a workaround for the old routing model. With consistent tap behavior, the user taps a different day on the calendar instead.

    **Widget tests** (`day_detail_sheet_test.dart`):
    - Empty past day shows "I had my period" button
    - Empty future day shows view-only text, no action buttons
    - Predicted past day shows prediction card + "I had my period"
    - Predicted future day shows prediction card + view-only message
    - Period day without entry shows "Add symptoms" + "Remove this day"
    - Period day with entry shows data + "Edit" + "Clear symptoms" + "Remove this day"
    - Mock CalendarViewModel + PeriodRepository for all tests
