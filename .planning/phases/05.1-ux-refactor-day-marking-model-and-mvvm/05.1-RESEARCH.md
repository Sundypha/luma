# Phase 05.1: UX Refactor — Day-Marking Model & MVVM — Research

**Researched:** 2026-04-06
**Domain:** Flutter MVVM architecture, UX refactor, domain algorithm, database migration
**Confidence:** HIGH

## Summary

Phase 05.1 is an internal refactor that replaces explicit "start period"/"end period" user actions with a day-marking toggle model, and restructures the presentation layer from ad-hoc StatefulWidget state into MVVM with ChangeNotifier ViewModels. The codebase is small (17 app Dart files, 9 domain, 9 data) and the refactor is well-scoped by the INVESTIGATION.md design document.

No new dependencies are required. Flutter's built-in `ChangeNotifier` + `ListenableBuilder` provide the full MVVM wiring. Drift 2.28's `TableMigration` API handles the v2→v3 schema migration (closing open periods, making `endUtc` non-nullable). The day-marking span derivation algorithm is pure domain logic — six well-defined operations (create, extend, merge, shrink, split, delete) each fully unit-testable without Flutter or a database.

**Primary recommendation:** Implement bottom-up (domain pure functions → repository methods → migration → ViewModels → Views → dead code removal) to ensure every layer is testable before wiring the next.

### Pattern 1: ViewModel as ChangeNotifier

**What:** Each screen gets a ChangeNotifier ViewModel that subscribes to repository streams, exposes read-only state, and provides command methods.

**When to use:** Every screen that currently uses StreamBuilder or manages logic in State.

**Example (from Flutter official architecture guide):**

```dart
class CalendarViewModel extends ChangeNotifier {
  CalendarViewModel(this._repository, this._calendar) {
    _subscription = _repository.watchPeriodsWithDays().listen(_onData);
  }

  final PeriodRepository _repository;
  final PeriodCalendarContext _calendar;
  StreamSubscription<List<StoredPeriodWithDays>>? _subscription;

  // Exposed read-only state
  List<StoredPeriodWithDays> _periodsWithDays = const [];
  Map<DateTime, CalendarDayData> _dayDataMap = const {};
  PredictionResult _prediction = const PredictionInsufficientHistory(
    completedCyclesAvailable: 0, minCompletedCyclesNeeded: 2);

  Map<DateTime, CalendarDayData> get dayDataMap => _dayDataMap;
  PredictionResult get prediction => _prediction;

  void _onData(List<StoredPeriodWithDays> data) {
    _periodsWithDays = data;
    _recompute();
    notifyListeners();
  }

  void _recompute() {
    final storedPeriods = _periodsWithDays.map((p) => p.period).toList();
    final predResult = PredictionCoordinator().predictNext(
      storedPeriods: storedPeriods, calendar: _calendar);
    _prediction = predResult.result;
    _dayDataMap = buildCalendarDayDataMap(
      periodsWithDays: _periodsWithDays,
      prediction: _prediction,
      today: DateTime.now(),
      startingDayOfWeek: DateTime.monday,
    );
  }

  // Commands
  Future<void> togglePeriodDay(DateTime day) async { /* ... */ }

  @override
  void dispose() {
    _subscription?.cancel();
    super.dispose();
  }
}
```

Source: [Flutter Architecture Case Study — UI Layer](https://docs.flutter.dev/app-architecture/case-study/ui-layer)

### Pattern 3: ViewModel Lifecycle Management (without Provider)

**What:** Since this project doesn't use Provider, ViewModels are created and disposed manually. The parent StatefulWidget (TabShell) owns the ViewModels.

**When to use:** The shell / root widget that creates child ViewModels.

**Example:**

```dart
class _TabShellState extends State<TabShell> {
  late final CalendarViewModel _calendarVm;
  late final HomeViewModel _homeVm;

  @override
  void initState() {
    super.initState();
    _calendarVm = CalendarViewModel(widget.repository, widget.calendar);
    _homeVm = HomeViewModel(widget.repository, widget.calendar);
  }

  @override
  void dispose() {
    _calendarVm.dispose();
    _homeVm.dispose();
    super.dispose();
  }

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      body: IndexedStack(
        index: _tabIndex,
        children: [
          HomeScreen(viewModel: _homeVm),
          CalendarScreen(viewModel: _calendarVm),
        ],
      ),
    );
  }
}
```

**Why not InheritedWidget:** The widget tree is shallow (TabShell → 2 screens). Constructor injection is simpler, more explicit, and already the project pattern. InheritedWidget adds boilerplate for no benefit at this scale.

Source: [ChangeNotifier dispose](https://api.flutter.dev/flutter/foundation/ChangeNotifier/dispose.html), [StackOverflow: When does dispose get called?](https://stackoverflow.com/questions/79398399)

### Anti-Patterns to Avoid
- **Business logic in build():** Current code computes predictions and builds dayDataMap inside `CalendarScreen.build()`. Move to ViewModel.
- **Stream subscriptions in widget State:** `LoggingBottomSheet` manages its own `StreamSubscription`. ViewModels own subscriptions instead.
- **Stale snapshots:** `DayDetailSheet` receives `allData` once at construction. ViewModels provide live state.
- **Mode flags in a single widget:** `LoggingBottomSheet` has 5 modes via `_SheetCreateIntent`. Replace with focused single-purpose sheets.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Reactive state binding | Custom stream → setState | ChangeNotifier + ListenableBuilder | Built into Flutter, tested, recommended by official guide |
| Schema migration column change | Raw SQL ALTER TABLE | Drift `TableMigration` API | Handles SQLite's 12-step procedure, copies data correctly |
| ViewModel lifecycle | Manual addListener/removeListener | StatefulWidget initState/dispose owning ChangeNotifier | Guaranteed cleanup on widget removal |

**Key insight:** The entire MVVM infrastructure uses zero external packages. Flutter ships everything needed.

## Common Pitfalls

### Pitfall 1: Forgetting to call `notifyListeners()` after async operations
**What goes wrong:** ViewModel state updates but the UI doesn't rebuild.
**Why it happens:** Async methods modify state in `then()` or after `await` without calling `notifyListeners()`.
**How to avoid:** Every method that changes exposed state must end with `notifyListeners()`. Use a `finally` block pattern as shown in Flutter's Compass app.
**Warning signs:** UI is stale after repository operations.

### Pitfall 2: Not disposing ChangeNotifier
**What goes wrong:** Stream subscriptions leak, memory grows, and listeners fire on disposed notifiers.
**Why it happens:** Without Provider, `dispose()` is not automatic. The owning StatefulWidget must call it.
**How to avoid:** TabShell (StatefulWidget) creates ViewModels in `initState()` and calls `viewModel.dispose()` in `dispose()`. Each ViewModel cancels its own stream subscriptions in `dispose()`.
**Warning signs:** "setState called after dispose" errors in debug console.

### Pitfall 3: Day-marking merge/split edge cases
**What goes wrong:** Two periods remain when they should merge, or a split creates overlapping periods.
**Why it happens:** Adjacency check off-by-one (using `<` instead of `<=` on day boundaries), or forgetting to reassign day entries when merging.
**How to avoid:** Pure domain functions with exhaustive unit tests covering all six operations. Key edge cases:
- Mark day between two 1-gap periods → must merge all three (two periods become one)
- Unmark middle day of 3-day period → split into two 1-day periods
- Unmark single-day period → delete period AND orphan day entries
- Mark/unmark at month boundaries (Dec 31 → Jan 1)
**Warning signs:** Orphan day entries (periodId references deleted period), overlapping periods.

### Pitfall 4: Migration data loss on open periods
**What goes wrong:** Open periods (`endUtc = null`) lose data or get wrong end dates during v2→v3 migration.
**Why it happens:** Using a fixed date (e.g., today) for all open periods instead of per-period logic (last day entry date).
**How to avoid:** Migration first runs UPDATE with subquery: `SET endUtc = COALESCE((SELECT MAX(dateUtc) FROM day_entries WHERE periodId = periods.id), startUtc)` for null rows, THEN runs `TableMigration` to make column non-nullable.
**Warning signs:** Open period end date doesn't match the user's last logged day.

### Pitfall 5: Drift TableMigration with non-nullable column
**What goes wrong:** Migration crashes because `endUtc` column still has null rows when Drift tries to copy to non-nullable schema.
**Why it happens:** Drift's `TableMigration` creates a new table with the target schema (non-nullable endUtc), then does INSERT INTO SELECT. If old rows still have NULL, the insert fails.
**How to avoid:** Two-step migration within the same transaction: (1) `customStatement` to UPDATE all NULL endUtc rows to a computed value, (2) `TableMigration` to rebuild the table with non-nullable column.
**Warning signs:** "NOT NULL constraint failed" during migration.

### Pitfall 6: Breaking existing tests during incremental refactor
**What goes wrong:** Changing PeriodSpan.endUtc from nullable to non-nullable breaks every existing test that constructs `PeriodSpan(endUtc: null)`.
**Why it happens:** The domain model change ripples through the entire codebase.
**How to avoid:** Keep `endUtc` nullable in Dart (`DateTime?`) but enforce non-null at the repository/DB boundary. OR: do the migration + model change as a single atomic step with a test fixup pass. The INVESTIGATION.md suggests keeping it nullable in Dart but never null in practice after migration — this is safer for incremental refactoring.
**Warning signs:** 50+ test failures from a single type change.

### Pitfall 7: SymptomFormViewModel lifecycle for bottom sheets
**What goes wrong:** Bottom sheet ViewModel leaks or references a disposed notifier after sheet closes.
**Why it happens:** Bottom sheets have their own lifecycle separate from the parent widget tree.
**How to avoid:** Create SymptomFormViewModel fresh when showing the sheet, using a StatefulWidget wrapper inside the sheet that creates/disposes the VM. Don't share it with the parent.
**Warning signs:** Errors after closing and reopening symptom forms.

## Code Examples

### Day-Marking Domain Algorithm

The core algorithm for all six span operations:

```dart
// packages/ptrack_domain/lib/src/period/day_marking.dart

DateTime _utcDay(DateTime d) => DateTime.utc(d.year, d.month, d.day);
DateTime _nextDay(DateTime d) => d.add(const Duration(days: 1));
DateTime _prevDay(DateTime d) => d.subtract(const Duration(days: 1));

bool _isAdjacent(DateTime day, PeriodSpan span) {
  final s = _utcDay(span.startUtc);
  final e = _utcDay(span.endUtc!); // endUtc always non-null post-migration
  return _utcDay(day) == _nextDay(e) || _utcDay(day) == _prevDay(s);
}

bool _containsDay(DateTime day, PeriodSpan span) {
  final d = _utcDay(day);
  final s = _utcDay(span.startUtc);
  final e = _utcDay(span.endUtc!);
  return !d.isBefore(s) && !d.isAfter(e);
}

// Returns (periodBefore, periodAfter) that are adjacent to `day`
(StoredPeriod?, StoredPeriod?) _adjacentPeriods(
  DateTime day, List<StoredPeriod> periods) {
  final d = _utcDay(day);
  StoredPeriod? before, after;
  for (final p in periods) {
    if (_utcDay(p.span.endUtc!) == _prevDay(d)) before = p;
    if (_utcDay(p.span.startUtc) == _nextDay(d)) after = p;
  }
  return (before, after);
}
```

## Open Questions

1. **Should `PeriodSpan.endUtc` become non-nullable in Dart?**
   - What we know: INVESTIGATION.md says "no more open periods" — endUtc always has a value after migration. However, changing the Dart type from `DateTime?` to `DateTime` breaks ~50+ constructor calls across tests and production code.
   - What's unclear: Whether the cost of a big type-change is worth the safety benefit, or if enforcing non-null only at the repository boundary is sufficient.
   - Recommendation: Keep `DateTime?` in Dart but add a `PeriodSpan.closed()` factory that requires non-null endUtc. Gradually migrate callers. The DB enforces non-null via schema. This avoids a single massive breaking change and allows incremental refactoring within the phase's plans.

2. **Day entries under merged/split periods: reassign or delete?**
   - What we know: When merging two periods, one is absorbed. Its day entries must be moved to the surviving period. When splitting, a new period is created for the detached segment.
   - What's unclear: The exact transaction sequence when merge involves day entries with conflicting dates (same calendar day under two different periods).
   - Recommendation: During merge, reassign all day entries from the absorbed period to the surviving period within a single transaction. Use `ON CONFLICT REPLACE` or an explicit upsert to handle duplicate calendar-day entries (keep the one with more data, or the more recent one).

3. **FAB behavior during calendar tab**
   - What we know: INVESTIGATION.md says FAB becomes "Mark today". But on the calendar tab, the FAB should perhaps mark the selected day, not today.
   - What's unclear: Whether the FAB should be context-aware (home tab = mark today, calendar tab = mark focused day) or always "mark today."
   - Recommendation: Keep FAB as "Mark today" on both tabs for simplicity. Calendar day-marking happens via tap → day detail sheet → "I had my period" button. This is consistent with the INVESTIGATION.md design.
